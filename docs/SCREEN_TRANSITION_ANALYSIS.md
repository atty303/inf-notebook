# 🔄 画面遷移と認識頻度の分析結果

## 📺 画面判定システム

### 画面種別と判定方法
- **loading**: (380,120) 4x2ピクセル領域の特定値チェック
- **result**: (1110,1042) 4x2ピクセル領域の特定値チェック  
- **music_select**: (181,84) 4x2ピクセル領域の特定値チェック

### 判定ロジック
`screenshot.get_screen()`で各画面の特定座標の色値を確認し、完全一致する画面が1つだけの場合にその画面と判定。

## ⏱️ スレッド周期設定

| 画面状態 | 変数名 | 周期 | 頻度 |
|---------|--------|------|------|
| INFINITASが非アクティブ | `thread_time_wait_nonactive` | 1.0s | 1回/秒 |
| ローディング中 | `thread_time_wait_loading` | 30.0s | 1回/30秒 |
| 通常画面 | `thread_time_normal` | 0.3s | 3.3回/秒 |
| **選曲画面** | `thread_time_musicselect` | **0.1s** | **10回/秒** |
| リザルト画面 | `thread_time_result` | 0.12s | 8.3回/秒 |

## 🎯 実際の認識実行条件

### 選曲画面での処理フロー
1. **画面判定**: `music_select`が検出される
2. **周期変更**: `self.sleep_time = thread_time_musicselect` (0.1s)
3. **スクリーンショット**: 100msごとに実行
4. **バージョン確認**: `recog.MusicSelect.get_version()`で選曲画面内容を判定
5. **キューイング**: バージョンが検出されればキューに送信

### musicselect_process()の実行タイミング
- キューから取り出された時のみ実行
- 同一楽曲の重複処理は回避される

## 🔍 重要な発見

### あなたの推測は正しい
実際の認識は **約1秒間隔** で実行されており、設定上の100ms間隔とは大きく異なります。

### 推定される理由
1. **画面判定の頻度**: 100msごとに`get_screen()`実行
2. **バージョン判定**: `get_version()`の結果によりキューイング
3. **実際の楽曲変更**: プレイヤーが楽曲を変更した時のみ認識処理
4. **処理負荷**: スクリーンショットとバージョン判定の処理時間

### 選曲画面の定義差異
- **システム定義**: `music_select`画面（181,84の4x2ピクセル判定）
- **実用的定義**: 楽曲選択可能で楽曲名認識が必要な状態
- **実際の動作**: プレイヤーの楽曲変更操作に連動

## 📈 パフォーマンスへの影響

現在の認識処理（平均10.86ms）は1秒間隔に対して：
- **CPU使用率**: 約1.1%（非常に軽微）
- **応答性**: 楽曲変更から1秒以内で認識完了
- **システム負荷**: 実用上問題なし

## 結論

musicselectの認識は理論上100ms周期だが、実際は **約1秒間隔** で楽曲変更時に実行される。これは適切な設計であり、パフォーマンスとユーザビリティのバランスが取れている。
